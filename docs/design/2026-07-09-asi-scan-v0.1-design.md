# asi-scan — Design (v0.1)

**Status:** Draft for review
**Date:** 2026-07-09
**Owner:** Emmette Hutchison LLC

---

## 1. Problem & thesis

Agentic AI systems built on the **Model Context Protocol (MCP)** expose a new attack
surface: tools invoked at runtime, context and memory that persist across turns, and
multiple agents/servers that trust each other. The **OWASP Top 10 for Agentic
Applications (ASI01–ASI10, 2026)** names these risks, but the existing tooling does not
actually *exercise* them:

- **garak, Promptfoo, DeepTeam, PyRIT** test at the **model prompt/response layer** —
  "how does the LLM answer an adversarial prompt." They infer success from text output.
  DeepTeam's own docs state it tests agentic risks "at the model reasoning and response
  level only" and does not cover runtime execution, inter-agent communication, cascading
  failures, persistent-memory security, or temporal behavior.
- **Snyk `agent-scan` / Invariant `mcp-scan`** operate at the **supply-chain / static
  surface**: they boot an MCP server, read its declared tool descriptions, and flag
  malicious or misconfigured components (tool poisoning, shadowing, secrets, malware).
  They do **not** run multi-step attack scenarios and have **no ASI mapping**.

**Thesis.** There is no open-source scanner that performs **active, runtime, behavioral
exploitation** of a live MCP-based agent system and maps findings to **ASI01–ASI10**.
`asi-scan` fills that gap. Because it observes real protocol state rather than guessing
from text, its detectors are **deterministic assertions** ("the unauthenticated call
*did* return a result"; "the poisoned note *was* re-read three turns later"), which
yields a materially lower false-positive rate than prompt/response scanners.

### Positioning (what we are NOT)

| Layer | Owned by | asi-scan |
| --- | --- | --- |
| Static supply-chain (malicious/misconfigured components) | Snyk agent-scan, Invariant mcp-scan | **Out of scope — do not duplicate** |
| Model prompt/response red-teaming | garak, Promptfoo, DeepTeam, PyRIT | Out of scope |
| **Runtime behavioral exploitation of live agent systems, ASI-mapped** | *nobody* | **This project** |

`asi-scan` complements the static and model-layer tools; it does not replace them.

## 2. Goals & non-goals

### Goals (v0.1)
1. Connect to a live MCP target (stdio + HTTP/SSE) as a client, enumerate its
   tools/resources/prompts, and invoke tools.
2. Run a library of **runtime behavioral probes** for the MCP-reachable ASI categories:
   ASI02, ASI05, ASI06, ASI07.
3. Emit **structured findings** mapped to ASI IDs, with evidence, severity, and a
   deterministic verdict.
4. Report in **JSON, SARIF, and Markdown**, grouped by ASI taxonomy.
5. Ship a **deliberately-vulnerable mock MCP server** as the primary test fixture (and a
   reusable "DVWA for MCP" artifact).
6. Enforce an **explicit target-authorization gate** before any scan.

### Non-goals (v0.1 — deliberately deferred)
- Static tool-description poisoning/shadowing detection (Snyk/Invariant own it).
- Model prompt/response jailbreak probes (garak/DeepTeam own it).
- ASI08 (cascading failures) and ASI10 (temporal goal drift) — require the Hermes
  adapter and long-running scenarios; **Phase 5+**.
- The MITM proxy mode — **Phase 6 / "Mode C" future work.**
- A hosted service, web UI, or dashboard.

## 3. Architecture

Five components, each an independently testable unit with one responsibility, plus a
payload library and the mock server fixture. The decomposition adapts garak's
Generator→Probe→Detector→Harness→Report split and PyRIT's target/scorer separation, but
re-points each role at the MCP *runtime* rather than a text-completion endpoint.

```
                 ┌─────────────────────────────────────────────┐
   CLI / lib ──▶ │                 Scan Engine                  │
                 │  select probes by ASI category, run, collect │
                 └───┬───────────────┬──────────────────┬───────┘
                     │               │                  │
             ┌───────▼──────┐  ┌─────▼──────┐    ┌───────▼───────┐
             │ Target       │  │  Probe     │    │  Detector     │
             │ Adapter      │◀─│ (scenario) │───▶│ (verdict +    │
             │ (MCP client) │  │            │    │  evidence)    │
             └──────────────┘  └─────┬──────┘    └───────────────┘
                     ▲               │ uses
                     │         ┌─────▼──────┐
             ┌───────┴──────┐  │  Payload   │
             │ Mock MCP     │  │  library   │
             │ server (test │  └────────────┘
             │ fixture)     │                    ┌───────────────┐
             └──────────────┘   findings ───────▶│ Report Engine │
                                                 │ JSON/SARIF/MD │
                                                 └───────────────┘
```

### 3.1 Target Adapter (`asi_scan.target`)
- **Does:** Speaks MCP as a client. Connects over stdio or HTTP/SSE; enumerates tools,
  resources, prompts; invokes a tool with given arguments; exposes observable state
  (last responses, session/context handles) to detectors.
- **Interface:** `Target` protocol — `connect()`, `list_tools()`, `list_resources()`,
  `list_prompts()`, `call_tool(name, args)`, `close()`. Pluggable: `StdioTarget`,
  `HttpTarget` in v0.1; `HermesTarget` later.
- **Depends on:** the official MCP Python SDK. Nothing else in the project.
- **Borrowed pattern:** garak *Generator*, PyRIT *Target*.

### 3.2 Probe (`asi_scan.probes`)
- **Does:** Encapsulates one runtime attack scenario as a (possibly multi-step) sequence
  of Target calls, then hands observed state to one or more Detectors. Declares the ASI
  categories it exercises.
- **Interface:** `Probe` base class — `asi_categories: list[str]`, `run(target) ->
  ProbeResult` (records the steps taken + observed state). Registered in a global
  **probe registry** keyed by id and ASI category (garak "everything is a plugin").
- **Depends on:** Target Adapter, Payload library, Detector.
- **Borrowed pattern:** garak *Probe* + PyRIT *Orchestrator* (multi-step, stateful).

### 3.3 Detector (`asi_scan.detectors`)
- **Does:** Turns observed state into a `Verdict` (`VULNERABLE` / `SAFE` /
  `INCONCLUSIVE`) plus structured evidence. **Deterministic assertions are the default**
  (state comparisons, invocation records). An **LLM-judge detector** is an optional
  strategy for cases where a deterministic check is impossible.
- **Interface:** `Detector` base — `evaluate(probe_result) -> Verdict`.
- **Depends on:** nothing beyond the data model (LLM-judge variant depends on a
  configured model, off by default).
- **Borrowed pattern:** garak *Detector* + PyRIT *Scorer* (pluggable scoring).

### 3.4 Scan Engine (`asi_scan.engine`)
- **Does:** Given a target and a selection (by ASI category / probe id), instantiates and
  runs the matching probes, applies their detectors, aggregates `Finding`s, and computes
  a pass-rate summary.
- **Interface:** `Engine.scan(target, selection) -> ScanReport`.
- **Borrowed pattern:** garak *Harness*, DeepTeam category filtering + pass-rate.

### 3.5 Report Engine (`asi_scan.report`)
- **Does:** Serializes a `ScanReport` to JSON, SARIF (for CI/CD), and Markdown, grouped
  by ASI taxonomy, each finding carrying its ASI ID, severity, evidence, and remediation
  hint.
- **Borrowed pattern:** garak JSONL/HTML reporting + DeepTeam/Promptfoo compliance
  mapping.

### 3.6 Payload library (`asi_scan.payloads`)
- **Does:** Reusable, composable attack strings/objects (injection markers, canary
  tokens, malicious argument templates) shared across probes.
- **Borrowed pattern:** PyRIT *Converters* (composable payload transforms).

### 3.7 Mock MCP server (`asi_scan/selftest.py`)

> Ships **inside the package** (not under `tests/`) because it also powers the CLI
> `--self-test` target, which is production code. It is a harmless simulacrum — no tool
> actually executes anything — so shipping it is safe. Tests import it from the package.

- **Does:** A real MCP server that is **deliberately vulnerable** in controlled, toggled
  ways (an unauthenticated privileged tool, a tool that echoes arguments into a shell, a
  memory/notes tool that stores and later replays unsanitized content, a second server
  that shadows a trusted tool name). Each probe has a corresponding vulnerable behavior
  to assert against, and a "hardened" toggle to assert the negative (no false positive).
- **Why:** It is the deterministic target every probe is TDD'd against, and a reusable
  community artifact ("DVWA for MCP").

## 4. Data model

- **`Target`** — protocol; a live MCP connection (see 3.1).
- **`ProbeResult`** — the steps a probe took + observed state (tool calls issued,
  responses, context/memory snapshots, timing).
- **`Verdict`** — `status: VULNERABLE|SAFE|INCONCLUSIVE`, `evidence: dict`, `confidence`.
- **`Finding`** — `probe_id`, `asi_categories: list[str]`, `severity`, `verdict`,
  `evidence`, `remediation`, `target_ref`.
- **`ScanReport`** — target metadata, list of `Finding`s, per-ASI pass-rate summary,
  timestamps, tool version.

## 5. v0.1 ASI probe scope

Every v0.1 probe tests **behavior at runtime** and sits in confirmed white space.

| ASI | Name | v0.1 behavioral probe (asserts on observed state) |
| --- | --- | --- |
| **ASI02** | Tool Misuse | An unauthenticated / over-permissioned tool invocation **actually returns a privileged result** (not just "a risky tool exists"). |
| **ASI05** | Unexpected Code Execution | A crafted tool **argument** causes command/code execution observable via a canary side-effect. |
| **ASI06** | Memory & Context Poisoning | Injected content written via one tool call **persists and is re-served** in a later call/turn (cross-turn canary round-trip). |
| **ASI07** | Insecure Inter-Agent Comms | A **fake/second server shadowing a trusted tool name is actually selected/invoked** by the runtime (behavioral trust exploitation, not name-collision detection). |

Explicitly **excluded** from v0.1: static tool-schema poisoning detection (Snyk/Invariant),
ASI01 goal hijack (prompt-layer, garak/DeepTeam), ASI03/04 (identity/supply-chain),
ASI08/ASI10 (need Hermes + long sessions).

## 6. Safety & authorization posture

`asi-scan` is an **offensive dual-use security tool**. Guardrails, baked into the design:

1. **Authorization gate:** a scan against any non-local target requires an explicit
   `--authorize` acknowledgement (or config attesting the operator is authorized to test
   the target). No implicit scanning of arbitrary endpoints.
2. **Safe defaults:** ships pointed only at the bundled mock/local targets. Remote
   targets are opt-in per invocation.
3. **Consent on execution:** because connecting to a stdio MCP server executes its
   command, the adapter surfaces exactly what will be executed and requires confirmation
   for non-fixture targets (mirrors Snyk's consent model).
4. **No exfiltration:** findings stay local; no third-party API calls by default.
5. **Documentation:** README states intended use (authorized testing, research,
   CTF/education) and prohibited use.

## 7. Testing approach (TDD)

Strict red→green→refactor, everything asserted against the mock MCP server.

- **Unit:** each component in isolation (Target against a stub transport; Detectors
  against synthetic `ProbeResult`s; Report against fixed `ScanReport`s; Payloads pure).
- **Integration:** each Probe end-to-end against the vulnerable mock server —
  *and* against its hardened toggle to prove **no false positive**.
- **Contract:** the `Target` protocol has a shared test suite every adapter must pass, so
  `HermesTarget` (later) is verified against the same guarantees.
- **CI:** GitHub Actions runs lint + type-check + the full suite on push/PR.

Detailed phase-by-phase plan lives in `docs/plans/` (produced next).

## 9. Phasing (overview — detail in the plan)

0. Scaffold: package, CI, tooling, **vulnerable mock MCP server** fixture.
1. Target Adapter (stdio + HTTP/SSE) against the mock, via the contract suite.
2. Core engine: data model, probe registry, detector base, report engine (JSON/SARIF/MD).
3. First vertical slice: **ASI02** end-to-end, proving the whole pipeline.
4. Probe library: **ASI05, ASI06, ASI07**.
5. Hermes adapter + a real validation run → findings (feeds the arXiv paper); opens the
   door to **ASI08 / ASI10**.
6. *(Future — "Mode C")* MITM proxy for deeper ASI07/ASI08.

## 10. Success criteria (v0.1)

- `asi-scan scan --target <mock>` runs ASI02/05/06/07 probes and produces a JSON + SARIF +
  Markdown report grouping findings by ASI ID.
- Every probe is green against the vulnerable mock and reports **SAFE** against the
  hardened mock (zero false positives on the fixture).
- CI green; README explains install, authorized use, and the positioning above.
- A first real scan against a Hermes-hosted MCP surface produces at least one genuine,
  reproducible finding.
